DOM Based XSS

El XSS (Cross-Site Scripting) basado en el DOM es una vulnerabilidad fascinante porque, a diferencia de otros tipos de XSS, el ataque ocurre enteramente en el navegador del usuario. El servidor ni siquiera se entera de lo que está pasando.

Aquí te explico cómo funciona, por qué es distinto y cómo identificarlo.

  1. 1.

    1. ¿Qué lo hace especial?

En el XSS Tradicional (Reflejado o Persistente), el código malicioso viaja al servidor y este lo incluye en la respuesta HTML.

En el XSS basado en el DOM, el flujo es distinto:

  1. El usuario visita una URL con una carga maliciosa (payload).
  2. El servidor responde con un HTML legítimo y scripts estáticos.
  3. El JavaScript legítimo de la página lee el payload de la URL y lo ejecuta o lo escribe en la página de forma insegura.

  1. 2.

    2. El concepto de "Source" y "Sink"

Para entenderlo, debemos ver el código como una tubería:

  • Source (Fuente): Es de donde el script toma los datos controlados por el atacante.
  • Ejemplos: location.search, location.hash (lo que va después del #), o document.referrer.

  • Sink (Receptor): Es la función o propiedad donde se "inyecta" el dato. Si esta función ejecuta código o renderiza HTML, tenemos un problema.
  • Ejemplos peligrosos: eval(), setTimeout(), o propiedades como .innerHTML.

El Ataque:

Un atacante te envía este enlace: http://ejemplo.com/inicio#user=<img src=x onerror=alert('XSS')>

  1. Tu navegador carga la página legítima.

  1. El script de la página extrae el código de la imagen de la URL.

  1. Al asignarlo a .innerHTML, el navegador intenta renderizar la "imagen" rota, falla y ejecuta el alert.

  1. 4.

    4. ¿Por qué es difícil de detectar?

Muchos firewalls de aplicaciones web (WAF) no lo ven porque los datos que van después del símbolo # (el fragmento) no se envían al servidor. El ataque es invisible para los logs de seguridad del backend; sucede puramente en la memoria del navegador.

Cómo protegerse

  • Evita .innerHTML: Usa .textContent o .innerText. Estos no interpretan el texto como HTML, solo como texto plano.

  • Sanea las fuentes: Si debes usar datos de la URL, utiliza librerías de limpieza como DOMPurify.

  • Content Security Policy (CSP): Configura reglas que impidan la ejecución de scripts no autorizados o inline.
